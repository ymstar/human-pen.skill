# 实战案例：Before/After 对照

> 适配版本：v1.1 | 更新日期：2025-06

本文档展示AI原稿到人类质感文章的4层改写过程，每层改写都有明确标注。

---

## 案例一：Claude Code Skills 介绍

### 原文（AI风格）

> Claude Code v2.1 版本发布，1096次提交刷新记录。Skills 作为其核心功能，值得深入探讨。值得注意的是，Skills 不仅仅是简单的提示词模板，更是一个强大的工作流系统。综上所述，我们可以得出结论：Skills 在工作流引擎领域具有重要的创新意义，希望这对您有帮助。

---

### 第1层改写：删除AI连接词

**改动**：删除"值得注意的是"、"综上所述"、"希望这对您有帮助"

> Claude Code v2.1 版本发布，1096次提交刷新记录。Skills 作为其核心功能，值得深入探讨。Skills 不仅仅是简单的提示词模板，更是一个强大的工作流系统。我们可以得出结论：Skills 在工作流引擎领域具有重要的创新意义。

**本层要点**：
- 去掉套路化结尾
- 去掉了"值得注意的是"这种废话连接

---

### 第2层改写：替换书面词汇

**改动**："深入探讨"→"翻了个底朝天"，"不仅仅"→"不光是"

> Claude Code v2.1 发布，1096次提交直接刷新记录。我在第一时间把 Skills 翻了个底朝天，得出的结论可能和大多数人想的不太一样：
>
> **Skills 不光是自定义命令，不光是提示词模板，它是一套工作流编程语言。**

**本层要点**：
- "深入探讨"→"翻了个底朝天"（口语化）
- "值得注意的是"→直接删掉或改成"得出的结论可能和大多数人想的不太一样"
- 核心观点加粗突出

---

### 第3层改写：增加真实细节

**改动**：加入个人经历"I"、具体场景、打字错误、口语表达

> 最近 Claude Code v2.1 发布，1096次提交直接刷新记录。
>
> 我在第一时间把 Skills 系统翻了个底朝天，得出的结论可能和大多数人的理解不太一样：
>
> **Skills 不是自定义命令，不是提示词模板，它是一套工作流编程语言。**
>
> 很多人把 Skills 和 CLAUDE.md 混为一谈。确实，两者都是配置文件，但本质完全不同。

**本层要点**：
- 增加"I"（第一人称经历）
- 增加对比具体场景
- 用"混为一谈"制造共鸣

---

### 第4层改写：增加人味

**改动**：加入踩坑经历、有态度的表达、互动感

> 很多人把 Skills 和 CLAUDE.md 混为一谈。确实，两者都是配置文件，但本质完全不同。
>
> 我之前写过一个代码审查的 Hook，想在每次 Write 之前自动跑 lint。结果发现 Hook 只能拦截单个工具调用，没法编排多步骤流程，最后还是改成了 Skill。

**本层要点**：
- 加入踩坑经历（写Hook失败的例子）
- 有态度（"结果发现"、"最后还是"）
- 留白让读者思考

## 案例二：多Agent系统问题描述

### 原文（AI风格）

> 多Agent框架已经多得让人眼花缭乱，LangChain/LangGraph/CrewAI/AutoGen 随便挑一个，两小时就能跑出一个演示。然而值得注意的是，演示和生产之间存在着巨大差距。许多团队在 POC 阶段信心满满，但上生产后不到一周就被状态丢失、失败雪崩、协作死锁这三个问题困扰。综上所述，问题的关键在于编排层无法支撑复杂的协作需求。

---

### 第1层改写：删除AI连接词

**改动**：删除"值得注意的是"、"综上所述"

> 多Agent框架已经多得让人眼花缭乱，LangChain/LangGraph/CrewAI/AutoGen 随便挑一个，两小时就能跑出一个演示。然而演示和生产之间存在着巨大差距。许多团队在 POC 阶段信心满满，但上生产后不到一周就被状态丢失、失败雪崩、协作死锁这三个问题困扰。问题的关键在于编排层无法支撑复杂的协作需求。

---

### 第2层改写：替换书面词汇+简化句式

**改动**：
- "多得让人眼花缭乱"→"多到挑花眼"
- "存在着巨大差距"→"隔着一道深沟"
- "不到一周就被...困扰"→"不到一周就被打崩"

> 2026年了，Multi-Agent 框架多到挑花眼。
>
> LangChain、LangGraph、CrewAI、AutoGen，随便挑一个，两小时跑出一个"看起来很智能"的演示。
>
> **但演示和生产之间，隔着一道深沟。**
>
> 我见过太多团队 POC 阶段信心满满，上生产后不到一周就被状态丢失、失败雪崩、协作死锁这三个隐藏瓶颈打崩。

**本层要点**：
- 把复杂句式拆成短句
- 用视觉化表达（"挑花眼"、"一道深沟"）
- 加粗核心观点

---

### 第3层改写：增加真实场景和情绪

**改动**：加入真实踩坑经历、情绪词、具体时间

> 说一个我踩过的真实坑。
>
> 去年带团队做一个"AI内容创作流水线"：选题Agent → 写作Agent → 审核Agent → 发布Agent。
>
> 听起来逻辑清晰，4个Agent各司其职。
>
> 跑起来第一天，正常。第二天，正常。
>
> **第三天早上，发现整个流水线停了。**

**本层要点**：
- "说一个我踩过的真实坑"直接开场
- 具体描述流水线设计
- 时间线制造悬念
- 用"打崩"这种口语化表达

---

### 第4层改写：增加判断和态度

**改动**：加入明确判断、有态度的结尾

> 查日志，发现写作Agent在凌晨2点调用外部LLM接口超时，整个Workflow卡在那里——没有重试，没有跳过，更没有告警。
>
> **重启之后呢？**
>
> 整个流水线从第一步重新开始。选题跑了两遍，审核收到了两份重复的初稿。
>
> 这不是代码写得烂，这是架构设计的问题。

**本层要点**：
- 凌晨2点这种细节增加真实感
- "这不是...是..."直接下判断
- 把技术问题讲成人话

## 案例三：Temporal vs LangChain 对比

### 原文（AI风格）

> LangChain 的定位是 Agent 编排框架，核心抽象是 StatefulGraph。这个抽象对单Agent场景非常友好，能够快速构建 RAG 流程、对话链路、多轮推理。值得注意的是，当场景变复杂，LangChain 的局限性就暴露了。Temporal 的核心特点是 Workflow 和 Activity 的分离，能够实现确定性重放和 Event Sourcing。综上所述，两者解决的问题根本不同。

---

### 第1层改写：删除AI连接词

**改动**：删除"值得注意的是"、"综上所述"

> LangChain 的定位是 Agent 编排框架，核心抽象是 StatefulGraph。这个抽象对单Agent场景非常友好，能够快速构建 RAG 流程、对话链路、多轮推理。当场景变复杂，LangChain 的局限性就暴露了。Temporal 的核心特点是 Workflow 和 Activity 的分离，能够实现确定性重放和 Event Sourcing。两者解决的问题根本不同。

---

### 第2层改写：口语化替换

**改动**：
- "核心抽象是"→"核心是"
- "非常友好"→"挺友好的"
- "局限性就暴露了"→"就不够用了"

> LangChain 的定位是 Agent 编排框架，核心是 StatefulGraph。这个抽象对单Agent场景挺友好的，快速构建 RAG 流程、对话链路、多轮推理都没问题。
>
> **但当场景变复杂，LangChain 就不够用了。**
>
> Temporal 呢？核心是 Workflow 和 Activity 的分离——确定性重放、Event Sourcing、原生支持长等待。

**本层要点**：
- 把"核心抽象是"口语化成"核心是"
- 用"挺友好的"替代"非常友好"
- 用"不够用了"替代"局限性暴露"
- 分段清晰，用对比结构

---

### 第3层改写：增加表格和具体对比

**改动**：加入表格、具体技术细节、有态度的表达

> LangChain 的定位是 Agent 编排框架，核心是 StatefulGraph。
>
> | 维度 | LangChain | Temporal |
> |------|-----------|----------|
> | 状态管理 | 内存或 Redis | Event Sourcing |
> | 容错 | try/catch | 内置重试 |
> | 等待机制 | 轮询/hack | 原生挂起 |
>
> **这不是谁更好，而是解决的问题根本不同。**
>
> LangChain 解决的是"怎么把 LLM 和工具链起来"。
>
> Temporal 解决的是"怎么可靠地编排可能运行几天的流程"。

**本层要点**：
- 加表格对比
- 用"这不是谁更好，而是..."表明立场
- 两句话总结核心差异

---

### 第4层改写：增加踩坑细节和人味

**改动**：加入代码示例、个人判断、有温度的结尾

> LangChain 的定位是 Agent 编排框架，核心是 StatefulGraph。
>
> 但当场景变复杂，LangChain 的局限就暴露了：
>
> ```python
> # LangChain 的状态存在内存
> # 重启后？丢了。
> 
> # LangChain 的容错靠 try/catch
> # 失败后？看你怎么写代码。
> ```
>
> Temporal 不一样：
>
> ```python
> # Activity 失败？自动重试
> # Worker 崩溃？Workflow 自动恢复
> # 想等3天？Workflow.sleep() 挂起，不占资源
> ```
>
> **我的判断：Temporal 适合对确定性要求高的业务场景，Skills 适合需要 AI 参与决策的工作流。两者不是替代关系，在复杂系统中很可能共存。**

**本层要点**：
- 用伪代码展示本质差异
- 用"I"表明个人判断
- 给出明确结论而不是"仁者见仁"

## 改写层级速查

| 层级 | 主要改动 | 难度 |
|------|----------|------|
| 第1层 | 删除AI连接词（此外、值得注意的是等） | ⭐ 简单 |
| 第2层 | 口语化替换（深入探讨→仔细看） | ⭐⭐ 中等 |
| 第3层 | 增加真实细节和个人经历 | ⭐⭐⭐ 较难 |
| 第4层 | 增加判断、态度、人味 | ⭐⭐⭐⭐ 难 |

**实用建议**：
- 先做第1、2层，60%的AI味就没了
- 第3、4层需要更多时间和创意积累
- 可以用黑名单扫描工具先跑一遍第1层
