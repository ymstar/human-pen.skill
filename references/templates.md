# 四平台文章骨架模板

> 适配版本：v1.1 | 更新日期：2025-06

本文档提供知乎、公众号、小红书、微博四个平台的文章结构模板，每个模板都包含具体示例。

---

## 一、知乎模板

### 结构：3行亮观点 → 背景踩坑 → 技术细节+代码 → 验证结果 → 明确判断（不要展望）

### 模板说明

| 部分 | 位置 | 要求 |
|------|------|------|
| 亮观点 | 开头3行 | 一句话表明核心立场，要反直觉或出人意料 |
| 背景踩坑 | 第1-2段 | 用真实经历/数据铺垫为什么要解决这个问题 |
| 技术细节 | 主体部分 | 含代码、原理、对比，不要只讲概念 |
| 验证结果 | 结尾前 | 展示实际效果，数据说话 |
| 明确判断 | 最后 | 直接说结论，不要"仁者见仁"或展望未来 |

### ❌ 知乎模板禁忌
- 结尾写"希望对您有帮助" → 直接删
- 结尾写"以上就是全部内容" → 直接删
- 结尾写展望/畅想未来 → 改成明确结论
- 开头废话铺垫 → 直接亮观点

### 技术内容示例

**主题：Claude Code Skills 不是提示词模板**

---

**【开头 - 3行亮观点】**

很多人把 Claude Code 的 Skills 和 CLAUDE.md 混为一谈。确实，两者都是配置文件，但本质完全不同：

> **Skills 不是自定义命令，不是提示词模板，它是一套工作流编程语言。**

---

**【背景踩坑】**

我之前写过一个代码审查的 Hook，想在每次 Write 之前自动跑 lint。结果发现 Hook 只能拦截单个工具调用，没法编排多步骤流程，最后还是改成了 Skill。

这就是 Skills 和 Hooks 的本质区别——Hooks 是拦截器，Skills 是执行单元。

---

**【技术细节 - 含代码】**

Skills 的设计精妙在三层能力的叠加：

```yaml
# 1. 按需加载 - 只注入当前任务需要的上下文
context: 
  scope: selective  # 不是一股脑塞给AI

# 2. 生命周期钩子 - 在工具调用前后插入逻辑
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./validate.sh"

# 3. 子Agent模式 - 完全隔离的子进程
context: fork
agent: Explore
```

对比一下 CLAUDE.md：

```yaml
# CLAUDE.md - 项目宪法，每次对话都加载
project_name: xxx
coding_style: xxx
```

CLAUDE.md 是"无条件注入"，Skills 是"按需激活"。

---

**【验证结果】**

我用这套模式跑了3个月：

- 上下文占用：从平均 180K 降到 45K
- 任务完成率：提升 23%
- Token 消耗：节省约 40%

---

**【结尾 - 明确判断】**

**结论：Temporal 适合对确定性要求高的业务场景，Skills 适合需要 AI 参与决策的工作流。两者不是替代关系，在复杂系统中很可能共存。**

## 二、公众号模板

### 结构：悬念标题 → 痛点共鸣 → 分段讲解（每段≤3行）→ 互动收尾

### 模板说明

| 部分 | 要求 |
|------|------|
| 标题 | 悬念感强，让人想点进来 |
| 痛点开头 | 前3行要引起共鸣，让人觉得"说的是我" |
| 分段讲解 | 每段≤3行，用emoji分隔，多配图 |
| 互动收尾 | 引导评论/点赞/收藏，不要太正式 |

### ❌ 公众号模板禁忌
- 标题太干/太学术 → 要有悬念/情绪
- 段落太长 → 每段不超过3行
- 太书面化 → 多用语气词、叠词
- 结尾太正式 → "感谢阅读" → 改成"有问题评论区见"

### 技术内容示例

**主题：多Agent系统为什么"看起来行，跑起来崩"**

---

**【悬念标题】**

我亲手跑崩了一个多Agent系统，才明白这个道理

---

**【痛点共鸣】**

2026年了，Multi-Agent 框架多到挑花眼。

LangChain、LangGraph、CrewAI、AutoGen，随便挑一个，两小时跑出一个"看起来很智能"的演示。

**但演示和生产之间，隔着一道深沟。**

我见过太多团队 POC 阶段信心满满，上生产后不到一周就被打崩。

---

**【分段讲解】**

**第1个坑：状态丢失**

🔹 内存里的状态，重启就没了
🔹 对话一长，token 烧得心疼
🔹 AI 推理天然有随机性

---

**第2个坑：失败雪崩**

🔹 A调B，B调API
🔹 API 超时 → B报错 → A崩 → 全链路停摆
🔹 3个Agent串行，整体成功率可能比单Agent还低

---

**第3个坑：协作死锁**

🔹 A要B的结果，B也要A的结果
🔹 等人工审批3天？靠轮询？那服务器资源白养着

---

**【结尾 - 互动引导】**

**核心就一句话：不是模型不够聪明，是编排层撑不住。**

你们公司现在用的是哪个框架？遇到过这些问题吗？

## 三、小红书模板

### 结构：吸睛标题+emoji → 场景导入 → 干货要点（分段+emoji）→ 互动引导+话题标签

### 模板说明

| 部分 | 要求 |
|------|------|
| 标题 | 公式：人群+痛点+数字+感叹，emoji点缀 |
| 场景导入 | 痛点/惊喜/身份认同，3秒抓眼球 |
| 干货要点 | 分段+emoji，重点加粗，配图穿插 |
| 互动引导 | "你们觉得呢？""评论区见" |
| 话题标签 | 3-8个精准标签 |

### ❌ 小红书模板禁忌
- 标题没有利益点 → 要说清楚看完能得到什么
- 大段文字 → 每段≤3行
- 纯干货没情绪 → 要有态度、emoji、互动
- 标签乱堆 → 精准不堆砌

## 四、微博模板

### 结构：核心观点一句话 → 展开说明（可选）→ 话题标签

### 模板说明

| 部分 | 要求 |
|------|------|
| 开头15字 | 必须抓眼球：疑问/数据/热点/利益点 |
| 核心观点 | 一句话说清，有态度 |
| 展开（可选） | 短微博不用，长微博分段+配图 |
| 话题标签 | 1个大话题+2-3个精准小话题 |
| 互动引导 | 引导评论 |

### ❌ 微博模板禁忌
- 开头废话 → 直接亮观点
- 标题党 → 标题和内容要一致
- 堆砌话题 → 精准不堆砌
- 无互动 → 要引导评论

**长微博示例（深度内容）**

---

为什么说多Agent系统需要 Temporal 而不是 LangChain？

说一个我踩过的真实坑 👇

---

【问题场景】

做了一个4步流水线：选题→写作→审核→发布

第三天整个停了——因为写作Agent凌晨2点调用外部接口超时，流程卡住。

**重启之后发生了什么？**

从第一步重新开始，选题跑了两遍，审核收到两份重复初稿。

---

【根本原因】

内存里的状态天然会丢。
外部调用天然会超时。
Agent之间天然有依赖。

**这三个问题不是更好的 prompt 能解决的。**

---

【LangChain vs Temporal】

| | LangChain | Temporal |
|--|---------|----------|
| 状态 | 内存/hack | Event Sourcing |
| 容错 | try/catch | 内置重试 |
| 等待 | 轮询/hack | 原生挂起 |

---

【结论】

Temporal 解决的是"可靠编排可能运行几天的流程"。

LangChain 解决的是"怎么把LLM和工具链起来"。

解决的问题根本不同，别混为一谈。
