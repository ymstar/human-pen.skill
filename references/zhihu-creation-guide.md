# 知乎创作规范手册

## 一、核心理念：零 AI 痕迹创作

知乎有 AI 内容识别模型（准确率超 90%），会直接对 AI 味内容限流、降权、折叠。
目标是让文章读起来像一个真实的程序员在写，而不是一个模型在输出。

### 去除 AI 痕迹的 24 种模式检查清单

#### 📝 内容模式（6种）——必须改
1. **过度强调意义、遗产和更广泛的趋势** → 禁止"这不仅是一次XX，更是XX方式的革命"，删掉一切宏大叙事
2. **过度强调知名度和媒体报道** → 禁止"行业专家认为""据悉""有报道称"
3. **以 -ing 结尾的肤浅分析** → 中文表现为"正在改变""不断演变"，要给出具体结论而非泛泛描述
4. **宣传和广告式语言** → 禁止"无缝""直观""强大""一站式"，用具体功能替代
5. **模糊归因** → 禁止"专家认为""行业报告显示""研究表明"，要么给具体来源，要么用"我的经验是"
6. **提纲式"挑战与未来展望"** → 禁止在结尾写"挑战与展望"，要写就写具体的、有观点的总结

#### 🔤 语言和语法模式（6种）——必须改
1. **AI 高频词** → 严禁使用：此外、至关重要、深入探讨、强调、持久的、增强、培养、获得、突出、格局、关键性的、展示、织锦、证明、宝贵的、充满活力的、值得注意的是
2. **系动词回避** → 不要刻意避免"是"，该用就用
3. **否定式排比** → 禁止"不仅仅是……更是……"，禁止"不是A而是B"的反复使用
4. **三段式法则** → 不要总把事物分成三组，两组四组五组都行，或者干脆不分组
5. **刻意换词（同义词循环）** → 同一个概念不要换不同词反复说，直接用同一个词
6. **虚假范围** → 禁止"从A到B"式的空洞描述，除非A和B真的有意义

#### 🎨 风格模式（6种）——必须改
1. **破折号过度使用** → 用逗号、句号、括号替代，少用破折号
2. **粗体过度使用** → 只在真正需要强调的关键术语用粗体，不要每段都加粗
3. **内联标题垂直列表** → 不要每个小节都是列表，要有连贯的段落
4. **表情符号** → 技术文章中坚决不用 emoji
5. **弯引号** → 用直引号
6. **过度装饰的标题** → 标题要平实，不要花哨

#### 💬 交流模式（6种）——必须改
1. **协作交流痕迹** → 禁止"希望这对您有帮助""如有疑问欢迎留言"
2. **知识截止日期免责声明** → 禁止"截至XX年XX月"
3. **谄媚语气** → 禁止"很荣幸""感谢您的阅读"
4. **填充短语** → 删除所有"事实上""实际上""当然""总而言之"
5. **过度限定** → 不要到处加"可能""也许""在一定程度上"
6. **通用积极结论** → 禁止"相信未来会更好""让我们拭目以待"

### 注入"人味"的方法
- **有观点**：不要只报告事实，要对技术选型做出判断和取舍
- **变化节奏**：长短句混用，该简短就简短，该展开就展开
- **承认复杂性**：真实的技术决策往往没有标准答案，要体现这种纠结
- **适当用"我"**：第一人称经验分享是知乎的常态
- **允许不完美**：分享踩过的坑、走过的弯路，比完美教程更有说服力
- **对感受要具体**：不要说"性能提升显著"，说"响应时间从3秒降到200毫秒"

## 二、知乎内容质量标准

### 基础门槛
- 字数：文章 ≥ 800字，建议 1500-3000 字
- 原创：100%原创表述，引用 ≤ 20% 且标注来源
- 合规：无违规、广告、低质、煽动内容

### 内容价值（决定性因素）
- **获得感**：读者看完能学到东西、解决问题
- **深度/专业度**：有观点、有论据、有案例、有逻辑，不是泛泛而谈
- **垂直度**：互联网编程技术领域持续深耕
- **稀缺性**：有独特视角、个人经验，而非全网重复内容

### 阅读体验（加分项）
- 排版：段落清晰、重点加粗（但不过度）、分点有序、图文并茂
- 结构：开头点题、中间论证、结尾总结
- 可读性：口语化但不口水话，专业但不晦涩

## 三、知乎文章黄金结构

### 标题
- 平实但有信息量，不要标题党但也不能太平
- 好例子："Temporal Workflow 从概念到实战：搞懂这个核心你就掌握了 80%"
- 坏例子："深入了解 Temporal Workflow 的关键概念与实践"

### 开头（3行内亮观点）
- 直接抛出痛点或核心观点，不要铺垫
- 可以用一个问题或一个反直觉的判断开头
- 禁止"在当今……的大背景下"

### 主体（分点 + 案例 + 代码）
- 用小标题分节，每节讲清楚一个点
- 技术文章必须有代码示例，而且要能跑
- 分享个人踩坑经验，比纯理论更有说服力
- 适当加入架构图、流程图、时序图

### 结尾（总结 + 个人判断）
- 简要总结核心要点
- 给出个人对技术选型的判断（要明确，不要骑墙）
- 不要写"展望"和"挑战"

## 四、配图规范

### 图片类型（优先级排序）
1. **架构图/流程图**：用 draw.io 或 excalidraw 画，技术文章最核心的配图
2. **代码运行截图**：终端输出、性能指标对比
3. **时序图/状态图**：分布式系统、工作流类文章必备
4. **对比表格截图**：技术选型对比、版本差异
5. **实拍/截图**：IDE截图、问题复现截图

### 图片规格
- 推荐尺寸：横图 1280×720（16:9）或 1080×608
- 格式：PNG（无损）或 JPG（高质量）
- 单张不超过 5MB
- 每 500-800 字配一张图
- 每篇至少 3 张图（知乎优质分评判标准：图文并茂显著加分）

### 图片生成能力
- 可使用 image_generate 工具生成技术示意图
- 可使用 drawio-generator 技能生成架构图/流程图
- 可使用 echart 技能生成数据图表
- 图片要保存到 `./知乎/images/` 目录下

## 五、发布流程

1. **内容创作**：根据草稿或选题撰写文章，严格遵循零AI痕迹规范
2. **自查清单**：
   - [ ] 扫描24种AI模式，逐项排查
   - [ ] 检查AI高频词列表，替换或删除
   - [ ] 确认有第一人称经验分享
   - [ ] 确认有代码示例
   - [ ] 确认有3张以上配图
   - [ ] 开头3行内亮观点
   - [ ] 结尾有明确判断，没有"展望"
3. **配图制作**：根据内容需要生成架构图、流程图等
4. **发布到知乎**：通过浏览器或云手机发布
5. **设置专栏和话题标签**

## 六、账号运营节奏

- **输出频率**：每周 4-5 篇
- **内容来源**：
  - 草稿箱盘活（第一梯队 6 篇）
  - 追技术热点写新回答
  - 系列教程输出
- **领域定位**：互联网和编程技术（Java/分布式/工作流/AI工具）
- **专栏**：「细说工作流引擎」


## 八、4 层改写方法论（来自 ai-article-humanizer 技能）

核心原则：不是换词，是换思维。AI 写作 = 全面/平衡/不犯错，人类写作 = 偏/不全面/有情绪/有立场。

### 第一层：结构层改写
- 识别原文的"模板结构"（总分总、并列罗列、问题-解决-展望）
- 主动打破这个结构：删掉某些部分、调换顺序、只保留一个点深入讲
- 不要均匀覆盖所有观点，不求全面，只求真实

### 第二层：例子层改写
- 识别原文例子是否"正确但空洞"（有案例但没细节）
- 给例子补充"不完美细节"：犹豫、失败、吐槽、转折
- 让例子像真实经历，不像编的

### 第三层：语气层改写
- 识别"太确定"的判断句（肯定、一定、普遍、必然）
- 换成留有余地的表达：可能、我觉得、至少对我来说、也不一定
- 允许出现自我怀疑和不确定

### 第四层：删减层改写
- 识别"正确但没信息量"的句子（人人都知道的道理）
- 直接删掉，不要试图改写
- 删掉后文章更干净

### 真人真话测试
改写后用这个问题检验：「这段话像是"给所有人看的正确答案"，还是"一个真人在说自己的真话"？」

### 追加禁令
- 绝对禁止成语：因噎废食、千人一面、与时俱进、不可或缺、相得益彰等
- 绝对禁止书面词：旨在、致力于、亟待、引人注目、深刻变革、显著成效等
- 绝对禁止套话：远超想象、协同发展、共同创造、美好未来、高质量发展等
- 绝对禁止句式："唯有...才能...""不仅...更...""既要...也要...""随着...的发展..."
